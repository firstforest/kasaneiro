# かさねいろ(kasaneiro)実装計画

[docs/note/](note/00-overview.md) の調査結果に基づく実装計画。作成日: 2026-07-04。

**計画の設計方針: 試行錯誤の速度を最優先する。** 調査で繰り返し確認された通り、このプロジェクトの品質はアルゴリズムの選択ではなく**パラメータ調整の反復回数**で決まる([01](note/01-fluid-simulation.md) §1「パラメータ調整が見た目を大きく左右する」、[05](note/05-rust-feasibility.md) §7「調整作業の大半=シェーダーパラメータいじり」)。そこで通常のロードマップに対し、次の2点を構造として組み込む:

1. **最初のマイルストーン(M0)を「実験ハーネス」の構築に充てる** — シミュレーション本体より先に、調整ループを回す道具(ホットリロード・パラメータパネル・デバッグ表示・ストローク再生)を作る
2. **各マイルストーンの完了条件を「目で見て判定できる体験」で定義する** — 数値目標ではなく「2色が馴染んで綺麗なグラデーションになるか」のような体感チェックにする。判定できないものは作りすぎのサイン

---

## 1. 試行錯誤ループの設計(このプロジェクトの心臓部)

理想の調整ループは以下で、**1周を数秒以内**に収めることが M0 の目標:

```
パラメータを1つ動かす(スライダー or WGSL編集)
  → 同じストロークが自動で再描画される(ストローク再生)
  → 前の結果と見比べる(スナップショット比較)
  → 良ければプリセット保存、悪ければ戻す
※ どこにも再コンパイル・再起動・手描きのやり直しが入らない
```

これを支える6つの装備。M0〜M1 で段階的に整備する:

| # | 装備 | 内容 | 効果 |
|---|---|---|---|
| H1 | **WGSLホットリロード** | シェーダーをファイルから実行時ロード。ファイル監視(notify クレート)で保存即再コンパイル。コンパイルエラーは画面にオーバーレイ表示し、直前の正常なパイプラインで続行(クラッシュさせない) | シェーダー編集で Rust 再コンパイル不要に。**調整作業の大半をカバー** |
| H2 | **パラメータパネル** | 全シミュレーションパラメータを単一の `SimParams` 構造体に集約し、そのまま uniform buffer として GPU へ。egui スライダーは構造体のフィールドと1対1。**パラメータ追加 = フィールド1行+スライダー1行+WGSL1行**の定型作業にする | スライダーはリアルタイム反映。蒸発率・沈着率・FlowOutward係数など全てをその場で動かせる |
| H3 | **プリセット保存/読込** | `SimParams` を serde で JSON 化して `assets/presets/` に保存・読込。**プリセットは git にコミットする**(「昨日の良かった設定」を失わないため)。顔料パラメータ(ρ, ω, γ, K, S)も同形式 | 試行錯誤の成果が蓄積される。悪化したら即戻せる |
| H4 | **デバッグ表示切替** | 表示モードを切り替えるセレクタ: 最終合成 / 水量 / 速度場 / 浮遊顔料 / 沈着顔料 / 紙ハイト / 湿りオーバーレイ(Rebelle の Show Wet 相当、青重ね)。実装は「どのテクスチャをどう可視化するか」を選ぶ表示用シェーダー1本 | 「なぜ滲まないのか」を見て診断できる。シェーダーデバッグ困難性への最大の対策 |
| H5 | **ストローク記録・再生** | 入力イベント列(位置・筆圧・時刻)を JSON で記録し、ボタン一発でキャンバスをリセットして同一ストロークを再描画。代表的なテストストローク(2色隣接、ウォッシュ、グレーズ)をリポジトリに同梱 | **同一入力でのA/B比較**が可能に。「手描きし直したら再現しない」を排除。将来の回帰チェックにも使える |
| H6 | **シミュレーション制御** | 一時停止 / 1ステップ実行 / 速度倍率 / キャンバスリセット。+ PNG スナップショット(ファイル名にプリセット名とタイムスタンプ) | にじみの進行をコマ送りで観察。before/after の証拠を残す |

**優先順位**: H1・H2 が最重要(M0 の完了条件)。H4・H6 は M1 序盤、H3・H5 は M1 で流体が動き始めてから(比較したいものができてから)で良い。

### CPU参照実装の位置づけ(やりすぎない)

[05](note/05-rust-feasibility.md) §7 の「CPU 参照実装+cargo test」は有効だが、全パイプラインの二重実装は過剰。**GPU の数値が疑わしくなったカーネル単位で後追い実装する**方針とする:

- 対象候補: 顔料移流(風上差分)、吸着/脱着(TransferPigment)、KM の R/T 合成式(§M3)
- KM 合成と K,S 逆算([03](note/03-layering-glazing-lifting.md) §3)は純粋関数なので最初から CPU 実装+テストで書き、WGSL へ移植して突き合わせる
- 流体本体(MoveWater)は数値の正しさより見た目なので、デバッグ表示(H4)で診断する方が速い

## 2. プロジェクト構成

```
kasaneiro/
├─ Cargo.toml
├─ src/
│  ├─ main.rs            # eframe 起動
│  ├─ app.rs             # egui パネル群(パラメータ・デバッグ表示・ツール選択)
│  ├─ gpu/               # wgpu 初期化、パイプライン管理、ping-pong、ホットリロード
│  ├─ sim/               # シミュレーション制御(パス順序、SimParams、テクスチャ群)
│  ├─ brush.rs           # ストローク → splat(位置補間、筆圧マッピング)
│  ├─ input.rs           # 入力抽象(trait)。マウス / ペン(egui Touch)→ (将来 wasm Pointer)
│  ├─ replay.rs          # ストローク記録・再生(H5)
│  └─ km.rs              # Kubelka-Munk 純関数(CPU参照、cargo test 対象)
├─ assets/
│  ├─ shaders/*.wgsl     # 実行時ロード(H1)。ビルドに埋め込まない
│  ├─ presets/*.json     # SimParams・顔料プリセット(H3、git管理)
│  └─ strokes/*.json     # テストストローク(H5、git管理)
└─ docs/
```

- ~~単一クレートで開始(ワークスペース分割は必要になってから)~~ → **2026-07-05 更新**: workspace 3 crate(km / pigment / paint-core)+バイナリへ分割する([refactoring.md](refactoring.md) R1)。mixbox 隔離を依存グラフで強制し、CPU 純粋部のテスト反復を速くするため
- **依存クレートは egui の対応バージョンを起点にロックステップで固定**(egui → egui-wgpu → wgpu → winit の順に合わせる。[05](note/05-rust-feasibility.md) §1 の注意)

wgpu / winit / egui + eframe + egui-wgpu / mixbox / notify(ホットリロード)/ serde + serde_json / image / rfd
(octotablet は M1.5 で不採用になった — §M1.5 の結果参照。筆圧は winit/egui 標準の Touch イベントで取れる)

## 3. マイルストーン

各マイルストーンは「完了条件(目で見て判定)」と「そこで回す試行錯誤」を持つ。順序は依存関係、期間の目安は集中作業日数の感覚値。

### M0 — 実験ハーネス(空アプリ+調整ループの土台)

**作るもの**: winit + wgpu + eframe のアプリ。egui の `PaintCallback` でキャンバス領域に storage テクスチャを表示。マウスドラッグでテクスチャに色を splat する compute パス1本。それを H1(WGSLホットリロード)+ H2(SimParams → uniform buffer → スライダー)で駆動する。

**完了条件**:
- マウスで描ける(splat が出る)
- WGSL を編集・保存すると**アプリを再起動せずに**描画結果が変わる。文法エラー時は落ちずにエラー表示
- スライダー(ブラシ半径など2〜3本)がリアルタイムに効く

**試行錯誤対象**: なし(道具作りの段階)。ここで egui/wgpu の統合パターンを固める。
**目安**: 3〜5日。[05](note/05-rust-feasibility.md) が言う「TS 案には無い先行投資」。ここを丁寧にやるほど以降が速い。

### M1 — wet-on-wet(要件1: 色が馴染むグラデーション)

Curtis 簡略版を **4つの小ステップ**に割り、各ステップで画面上の変化を確認しながら進める。テクスチャ構成は [00](note/00-overview.md) の推奨(水量+速度 / 浮遊顔料 / 沈着顔料 / 紙ハイト、512²、ping-pong)。

| ステップ | 内容 | 見て確認すること |
|---|---|---|
| M1a | 水の浅水層: 速度更新 → 発散の反復緩和(δ=−ξ·div)→ 水量移流。**濡れ領域マスク**(Curtis の wet-area mask。筆が通ったセルだけ水が動き、乾いた紙との境界は壁として扱う)を水テクスチャの a チャンネルに持つ。デバッグ表示 H4(水量・速度場・湿りオーバーレイ)もここで作る | 置いた水たまりが広がり、流れが見える(顔料なしで水だけ先に動かす)。**にじみがストローク領域の外へは広がらない** |
| M1b | 顔料層: 浮遊顔料の移流+吸着/脱着(TransferPigment)+蒸発。まず単顔料 | 水の流れに乗って色が広がる。水が減ると色が定着する |
| M1c | Mixbox 混色(LUT を WGSL 移植)+複数顔料(RGBA 4チャンネル=4顔料) | **2色を隣接して置くと、境界が濁らず馴染む(黄+青=緑)** |
| M1d | FlowOutward(エッジダークニング)+紙ハイトの3箇所適用([01](note/01-fluid-simulation.md) §6)。H3(プリセット)・H5(ストローク再生)をここで整備 | 乾いた縁が暗くなる。紙目に沿ったストリークが出る |

**完了条件(= プロジェクト最大の検証点)**: *先に置いた色の近くに別の色で描くと、馴染んで綺麗なグラデーションになる* — 要件1がマウス描きで体感できる。
**試行錯誤対象**: 蒸発率、発散緩和の ξ と反復回数、吸着/脱着率、FlowOutward の η とぼかし半径、紙ハイトの振幅。**ここが本計画で最も長く滞在する場所**であり、H1〜H6 が全て効いてくる。
**目安**: 実装2週間+調整は無期限(満足するまで)。

**M1 のリスクと退路**:
- 浅水方程式が不安定・高粘性に感じたら → Stam のセミラグランジアン移流に置換(Van Laerhoven 2005 の処方。[01](note/01-fluid-simulation.md) §3)。移流カーネル1本の差し替えで済むようシェーダーを分けておく
- にじみ境界の表情が物足りなければ → LBM(MoXi)換装は M4 の選択肢として保留。まず Curtis 系でパラメータを詰め切る

### M1.5 — 筆圧

マウスで wet-on-wet が成立してから、`input.rs` の trait 実装にペンを追加。筆圧 → 水量・顔料量・半径へのマッピングを H2 のスライダーで調整可能にする。

**完了条件**: ペンの筆入れ・筆抜きで水と顔料の量が変わり、ストロークに強弱が出る。
**結果(2026-07-05 完了)**: 計画していた octotablet は、Manager を作るとメッセージループがデッドロックしアプリが応答なしになる既知バグ([issue #18](https://github.com/Fuzzyzilla/octotablet/issues/18)。0.1.0 / git master とも再現)により**不採用**。実際には egui 標準の Touch イベント(winit が WM_POINTER から `GetPointerPenInfo` の筆圧付きで届ける)だけで筆圧が取れたため、追加依存なしで実装した。詳細は status.md の M1.5 行。

### M2 — 乾燥とレイヤー(要件2: グレージング)

[03](note/03-layering-glazing-lifting.md) §6 の共通アーキテクチャに従う:

- 湿レイヤーは常に1枚。**常時見える「乾かす」ボタン**(Fresco の UX 教訓 — メニューの奥に隠さない)
- 乾燥 = 定着パスを1回走らせて RGBA レイヤーに焼き込み、湿潤バッファを解放
- レイヤー合成はまず **multiply** で開始(KM 合成は M3)
- レイヤーパネル(表示/非表示、順序)を egui で
- Rebelle 式の安価な追加: Fast Dry(水だけ除去)/ Wet the Layer(再湿潤)は効果対コストが良いので入れる

**完了条件**: 乾かした色の上に別の色を描くと、下の層が動かず、重なりが透けて見える(要件2)。
**試行錯誤対象**: 定着パスでの dry shift(乾くと薄くなる)、焼き込み時の粒状感ゲート。
**目安**: 1〜2週間。

### M3 — 削り・顔料個性・KM合成(要件3+発色の仕上げ)

- **リフティングツール**([03](note/03-layering-glazing-lifting.md) §5 のレシピ): 沈着層→浮遊層への転送、ステイン床クランプ(`ω·gd_initial`)、紙ハイトでのマスク変調、剥がした顔料の縁への再沈着
- **完全消去ツール**を別ツールとして併設(Painter 方式の2ツール構成)
- 顔料パラメータ **ρ / ω / γ + K,S** をプリセット化(Curtis 論文の12顔料の実値表から開始)
- レイヤー合成を multiply → **KM の R/T 合成**へ置換(`km.rs` の CPU 実装をテストで検証してから WGSL 化。リニア色空間・sinh/cosh のオーバーフロー対策)

**完了条件**: 削って白(に近い部分)を作れる。ステイニング顔料は床が残り、粒状化顔料は紙目に溜まる、と顔料ごとの個性が見える。
**試行錯誤対象**: 顔料12種の効き、リフト率、KM 合成前後の見た目比較(H5 のストローク再生で multiply と A/B)。
**目安**: 2週間。

### M4 — 道具としての仕上げ+選択的高度化(任意)

必要性を感じた順に。**どれも M3 までの成果物が先に「使える」状態になってから**:

- 保存/読込(独自形式: レイヤー PNG 群+メタ JSON)、PNG 書き出し、undo(まずキャンバス全体スナップショット方式で十分。512² RGBA 数枚 ≒ 数MB/step)
- アクティブタイル最適化(512² 全面更新で 60fps を割った場合のみ。近代 GPU では当面不要の見込み — [04](note/04-implementation-stack.md) §4)
- 解像度引き上げ(1024²)、シミュレーション解像度と表示解像度の分離
- エッジダークニングの追加調整、実紙スキャンの紙テクスチャ差し替え
- LBM(MoXi)換装の実験、wasm/Web 版の試験ビルド

→ 2026-07-05: このうち**保存/読込・undo・アクティブタイル・解像度引き上げは M5〜M8 として具体化**した(下記)。

### M4.5 — 線画とハイライト

計画追加: 2026-07-05。下書き(鉛筆)→ 清書(ペン)→ 彩色 → ハイライトのワークフローを支える。**着手前に [refactoring.md](refactoring.md) の R1〜R4(workspace 化・Tool 階層 enum・パイプラインのテーブル化・UI 分割)を適用する** — ラスタツール追加で経路が分かれる前に土台を整えるため。**ラスタ実装**(専用テクスチャ) — ベクター化は §4 で先送り(ストローク記録 H5 が生入力を保持するので、後からベクター再構成する余地は残る)。M7 の保存形式が決まる前にデータ構成を確定させるため、M5 より先に置く。

| ステップ | 内容 | 見て確認すること |
|---|---|---|
| M4.5a | **線画テクスチャ+鉛筆/ペンツール**: 鉛筆・ペン各 r32float ×1(ping-pong 不要、read_write storage にブラシが直接描く)。ツール 5=鉛筆(グレー粒状線、紙ハイトで濃度変調、筆圧→濃さ)/ 6=ペン(濃色スムーズ線、筆圧→太さ)+各消しゴムモード(トグルで splat が減算に反転)。display.wgsl の合成で湿レイヤーの上に挿入(multiply/KM 両モード)。レイヤーパネルに下書き・清書の表示切替(並べ替え対象外・位置固定) | 鉛筆で下書きし、ペンでなぞって清書できる。下書きだけ非表示にして仕上げられる。鉛筆に紙目の粒状感が出る |
| M4.5b | **境界効果(透水率方式)**: ペン濃度から透水率 `perm = 1 − line_block×ペン濃度` を出し、①拡散(diffuse)の隣接流束 ②にじみ拡張(wet_expand の蓄積)③速度場(velocity)に掛ける。**ブラシの直接スプラットには掛けない**(線を跨ぐストロークは両側に置かれる=明示的な筆使いなら越えられる) | ペン線で囲った領域を塗っても水が線を越えない。線を跨いでストロークすると越える。`line_block=0` で従来どおり |
| M4.5c | **ハイライトレイヤー**: r32float ×1(不透明度 0..1)。ツール 7=不透明白ブラシ(流体シミュなし、筆圧→不透明度)+消しゴムモード。display.wgsl の合成の最後に `mix(色, 白, ハイライト)`(multiply/KM 両モードで最上段)。表示切替あり | 乾いた絵の上に白でハイライトを入れられ、下の層が一切動かない |
| M4.5d | **線画・ハイライトの多段 Undo/Redo**: 鉛筆・ペン・ハイライトのストローク(消しゴム含む)をレイヤーごとの履歴リストに保持し、**Undo = テクスチャをクリアして残りストロークを再ラスタライズ / Redo = 取り消し分を再適用**。流体シミュを通らないので再生は厳密に決定論的。**ストロークごとに実効パラメータ(半径・筆圧マッピング等)を保存**する — スライダーを変えた後の Undo/Redo で過去の線の形が変わらないため(H5 の「現在のパラメータで再生」とは目的が逆)。新ストロークで Redo 履歴は破棄。Ctrl+Z / Ctrl+Shift+Z。湿レイヤー(水彩)は対象外 — M6 の undo 1段のまま | 線を何本か描いて Ctrl+Z を連打すると1本ずつ戻り、Redo で1本ずつ復元される。ブラシ半径を変えてから Undo/Redo しても既存の線の形が変わらない |

**完了条件**: ペンの線で囲んだ領域を塗ってもはみ出さず、意図して線を跨ぐと越える。下書きを非表示にして清書+彩色+白ハイライトの一枚が作れる。線画は多段 Undo/Redo で気軽に引き直せる。
**試行錯誤対象**: `line_block` の効き(越えにくさと窒息感のバランス)、鉛筆の粒状感、ペンのエッジの硬さ、ハイライトの不透明度カーブ。
**目安**: 1〜1.5週間。

**M4.5d のリスクと退路**: ストローク数が増えて再ラスタライズが遅くなったら、Kストロークごとの周期チェックポイント(テクスチャ退避)を足して「最寄りのチェックポイント+差分再生」にする(§4 の先送り表)。512² r32float は 1MB/枚なので当面は全再生で足りる見込み。

### M5 — パレット(顔料の作成・編集・スポイト)

計画追加: 2026-07-05。固定4顔料([src/pigment.rs](../src/pigment.rs) の const)を「**自由に編集できる4スロット+無限にストックできるライブラリ**」に広げる。同時顔料数(テクスチャ4ch)は変えない — 選べる色の少なさはライブラリで、1枚の絵の多色化はレイヤーごとのパレット記録(下記③)で解決するため、チャンネル拡張は不要(§4 の先送り表参照)。

| ステップ | 内容 | 見て確認すること |
|---|---|---|
| M5a | **顔料パラメータの編集 UI 化**: ホバー表示 → カラーピッカー+ρ/ω/γ スライダー(平易な日本語ラベル)。per-顔料 `density`(ρ)と SimParams の `pigment_density` の用語を分離 | スライダーを動かすと顔料の性質(沈み方・剥がれ方・粒状感)が変わって見える |
| M5b | **ランタイムパレット**: `PIGMENTS` の const → 状態化。編集時に `latent_uniform` / `physics_uniform` を再計算して `write_buffer`(バッファは COPY_DST 済みなのでパイプライン再構築不要) | 好みの色・性質の顔料を作って描ける |
| M5c | **レイヤーごとパレット記録**: 「乾かす」の焼き込み時に現行パレットの latent をそのレイヤー専用に記録し、display.wgsl は層ごとの latent で発色(latents uniform をレイヤー分に拡張、8層×14 vec4 ≒ 7KB)。ρ/ω/γ は湿シミュ専用なのでグローバルのまま。**M5b と同じスライスで入れる**(これが無いと「顔料編集 → 過去の絵が遡って変色」バグ付きになる。原因: 表示が グローバル latents × 保存濃度 の毎フレーム発色のため) | **顔料を編集しても乾燥済みレイヤーの色が変わらない**。副効果: 乾かすたびにスロットを差し替えれば1枚の絵に 4色×レイヤー数 の顔料が使える |
| M5d | **パレット・ライブラリ**: `assets/palettes/*.json` 保存/読込(H3 と同じ流儀、git 管理)。ストローク記録(H5)にもパレットを保存(serde default で旧ファイルは現行パレット扱い。スロット番号しか持たない記録は再生時の色が変わり A/B 比較が壊れるため) | 昨日作った顔料を今日も使える。古いストローク記録が当時の色で再生される |
| M5e | **スポイト**: `snapshot()` の読み戻し経路で カーソル下1px の表示色を取得 → mixbox latent 化して選択スロットへ(表示色版。顔料構成を拾う版は §4 で先送り) | キャンバス上で混ざってできた色を拾って続きが描ける |
| M5f | **色単体ライブラリ**(計画追加 2026-07-09): 顔料1個(名前・色・ρ/ω/γ)を `assets/pigments/*.json` に保存/読込(palette_store と同じ流儀、git 管理)。保存名=顔料名の1本化(名前欄なし)。色ライブラリモーダル(FileModal)+左パネル「この色をとっておく」1クリック保存 | 作った1色だけを取っておき、後で任意のスロットへ呼び出せる |
| M5g | **パレット編集の動線改善**(計画追加 2026-07-09): 折りたたみの中身を「選択中スロットのみ編集」に縮小(ツールバーの色スウォッチが編集対象の切替を兼ねる)。パレット保存/読込・既定に戻すは4色見本チップ付きのパレットモーダルへ移設 | 編集対象がスウォッチ1クリックで切り替わり、パレット一覧が色見本で選べる |

**目安**: 1週間(M5f〜h は +2日)。

### M6 — スケールする土台(アクティブタイル・ビューポート・undo)

M8(2048²)の前提。**全面計算のまま解像度を16倍にしない**(順序を逆にすると、重くなった状態でタイル化をデバッグする羽目になる)。512² のまま実施する。

- **アクティブタイル**: 濡れタイル(+ブラシ周辺)のみ compute を dispatch。コストを「紙の広さ」比例から「濡れ面積」比例へ。splat.wgsl の全テクセル×全スプラットループもタイル限定で自然に解消。デバッグ表示(H4)にアクティブタイル可視化を追加
- **パン/ズーム**: キャンバスがウィンドウ(1100×780)より大きくなるため必須。egui の PaintCallback 内で表示変換
- **undo 1段**: ストローク開始時にテクスチャを退避し、直前ストロークのみ取り消し(複数段は §4 で先送り)。対象は湿レイヤー(線画・ハイライトは M4.5d の多段 Undo/Redo が既にあるため対象外)。Ctrl+Z は**最後の操作の種別で振り分ける統一履歴**にする(線を描いた直後は線が戻り、水彩ストロークの直後は湿レイヤーが戻る)

**完了条件**: 部分的なストロークでは計算がその近傍だけ走っている(タイル可視化で確認)。ズーム/パンしながら描ける。直前のストロークを取り消せる。
**目安**: 1〜2週間。

### M7 — 作品保存

2048² で時間をかけた絵を描き始める前に用意する(現状は PNG スナップショット=見た目のみ、ストローク記録=作り直し、しかなく**描きかけを翌日続ける手段がない**)。

- 独自形式1ファイル: 全シミュレーションテクスチャ+乾燥レイヤー+レイヤー構成+**各レイヤーのパレット(M5c)**+現行パレット+**線画(鉛筆・ペン)とハイライトのテクスチャ(M4.5)**。線画の Undo 履歴(ストローク列)はテクスチャがあれば復元不要なので保存しない(読込直後の Undo は効かない、で割り切る)
- 保存の trait 抽象(plan §4 の Web 版先送り条件)は維持

**完了条件**: 保存 → アプリ再起動 → 読込で、描きかけの状態(湿った絵の具も含む)から続きが描ける。
**目安**: 3〜5日。

### M8 — キャンバス 2048²

用途は「広い紙」(ブラシ・にじみはテクセル単位で従来のまま=絵の密度は据え置き)。この前提なら `diffuse_iters` や `edge_radius` などテクセル単位の距離・反復は据え置きでよく、**超線形のコスト増(拡散の反復爆発・ぼかし半径の2乗)を踏まない**。同じ見た目スケールを保つ高精細化(ブラシもテクセルで4倍)をやりたくなったら別問題として再検討。

- `CANVAS_SIZE` const → 起動時(または新規作成時)設定。正方形のまま 512/1024/2048 を選択式(試行錯誤は 512² で続けられる = 反復速度の原則を守る)
- VRAM 目安: 2048² rgba32float ≒ 67MB/枚、計 ~1GB(乾燥レイヤー8枚 = 537MB が支配的。窮屈なら `MAX_LAYERS` 削減で調整)
- 移流はセミラグランジアンで無条件安定のため解像度変更に対して安全。パラメータ再調整は最小の見込み

**完了条件**: 2048² で、濡れ面積が普通の筆致の範囲なら 60fps で描ける。
**目安**: 3〜5日(M6 が済んでいれば設定化が主作業)。

## 5. 最初の一歩

M0 の着手手順:

1. `cargo init` + eframe テンプレートでウィンドウ表示
2. egui-wgpu の `PaintCallback` でキャンバス領域に単色テクスチャ表示
3. compute パス1本(splat)+ マウス入力
4. WGSL のファイルロード+notify 監視+エラーオーバーレイ(H1)
5. `SimParams` → uniform buffer → スライダー2本(H2)

ここまで動けば M1a(水を動かす)に進める。
